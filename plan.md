# RAG、知识图谱与多智能体扩展计划

## 目标

在保留现有文件读取与中文分块能力的基础上，扩展出可复用的 RAG 链路、知识图谱链路和多智能体编排层。三者共享同一套文档读取、分块、元数据和错误处理底座。

## 总体原则

- 先统一数据契约，再接具体存储实现。
- 共享 ingestion 底座，不重复实现读取和分块。
- RAG、知识图谱和多智能体编排使用同一份 chunk 元信息。
- 保留容错、降级和并发控制。
- 所有输出都要能追溯回原文件和原始位置。

## 推荐结构

### 1. 基础层

负责把原始文档转换成标准化的中间数据。

- 输入：files 目录或上游上传文档
- 处理：文件读取、内容清洗、中文分块、元数据补全
- 输出：document、chunk、hash、source_span、processing_status

建议复用现有模块：

- extractor/filereader/read_file.py
- extractor/filereader/text_chunker.py
- extractor/filereader/document_processor.py

### 2. RAG 层

负责把 chunk 写入向量检索系统，并提供查询时的召回与重排。

- 入库流程：document -> chunk -> embedding -> vector store
- 查询流程：query -> vector retrieval -> metadata filter -> rerank -> answer
- 检索策略：优先支持混合检索，兼顾向量召回和关键词过滤

### 3. 知识图谱层

负责从 chunk 中抽取实体与关系，并写入图数据库。

- 抽取流程：chunk -> entity extraction -> relation extraction -> dedup -> graph upsert
- 图谱单元：entity、relation、evidence、source_chunk
- 回溯要求：每个实体和关系都要记录来源 chunk 和原文证据

### 4. 多智能体编排层

负责把读取、分块、RAG、KG 和回答生成拆成可协作的智能体，并统一调度。

- 文档理解智能体：负责读取、清洗和分块
- 检索智能体：负责向量检索、过滤和重排
- 图谱抽取智能体：负责实体识别、关系抽取和入图
- 规划智能体：负责任务拆解、路由和结果汇总
- 协调层：负责消息传递、状态管理、重试和并发控制

## 分阶段计划

### 阶段 1：统一数据模型

目标是定义后续所有模块都能复用的数据结构。

- 定义 document 对象
- 定义 chunk 对象
- 定义 entity 和 relation 对象
- 定义必须保留的元信息字段
- 定义处理状态和错误字段

交付物：

- 标准化数据契约说明
- chunk 元信息字段清单

### 阶段 2：升级基础 ingestion

目标是让现有读取与分块输出可直接供 RAG 和 KG 使用。

- 在现有 document processor 上补充标准化输出
- 为每个 chunk 增加来源文件、位置和 hash
- 保留批处理容错
- 保留 HanLP 可用时启用、不可用时降级的策略

交付物：

- 标准化 ingestion 输出
- 可追溯 chunk 数据

### 阶段 3：实现 RAG 主链路

目标是完成文档入库和检索闭环。

- 接入 embedding 生成
- 接入向量库写入
- 接入检索接口
- 增加可选 rerank
- 增加简单问答入口

交付物：

- 向量入库流程
- 检索流程
- 最小可用问答链路

### 阶段 4：实现知识图谱主链路

目标是完成实体关系抽取和图数据库写入闭环。

- 增加实体识别
- 增加关系抽取
- 增加实体去重和对齐
- 增加图数据库 upsert
- 保留证据片段和来源 chunk

交付物：

- 实体关系抽取流程
- 图谱写入流程
- 可回溯证据链

### 阶段 5：实现多智能体编排

目标是把各能力拆成可以协作的智能体，并形成统一调度。

- 定义智能体职责边界
- 定义任务路由和状态流转
- 定义智能体之间的输入输出格式
- 定义失败重试和降级策略
- 定义统一结果汇总方式

交付物：

- 多智能体编排流程
- 任务路由规则
- 状态管理和协作规范

## 多智能体设计（假设 Neo4j 与向量库已就绪）

### 角色定义

- 路由智能体：识别用户问题类型，决定走图谱检索、向量检索或混合检索。
- 图谱检索智能体：面向 Neo4j 执行关系与路径查询，返回结构化证据。
- 向量检索智能体：面向向量库执行语义召回，返回文本证据片段。
- 融合应答智能体：对两路结果去重、冲突消解并生成带来源的答案。

### 路由规则

- 关系型问题优先图谱检索。
- 解释型问题优先向量检索。
- 意图不明或复杂问题走混合并行检索。

### 协作流程

1. 路由智能体解析 query 并生成检索计划。
2. 图谱检索与向量检索并行执行。
3. 融合应答智能体完成证据对齐与打分。
4. 返回 answer、evidence 与 trace。

### 降级策略

- Neo4j 超时时自动降级为向量检索。
- 向量检索失败时回退图谱检索。
- 双路都低置信度时返回澄清问题而非强答。

### 阶段目标

- 首先打通单轮查询的稳定路由与混合检索。
- 其次补齐 evidence 与 trace 的可追溯输出。
- 最后优化路由准确率与端到端延迟。
